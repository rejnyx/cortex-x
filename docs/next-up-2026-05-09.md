## Why my recommendation is "A + C parallel"

1. **A is overnight-passive.** Once Day 0 supervised run is clean (~30 min), the burn-in runs *without operator attention*. Operator does 5-min daily reviews. **No focus cost on the implementation track.**
2. **C is the only one of B / B′ / C that is *truly independent of B (worktree supervisor)*.** Sprint 2.2 + 2.3 both touch `bin/steward/_lib/` and benefit from worktree isolation. Sprint 2.9 ships a *new* module tree (`bin/cortex/tools/`) that doesn't touch the steward critical path beyond a thin annotation-routing hook in `action-engine.cjs`.
3. **C compounds with A.** Sprint 2.9 tool annotations (`destructiveHint` / `readOnlyHint`) wire into Sprint 1.9.0 spec-verifier and Sprint 2.7 path-traversal hardening. **Burn-in data from A directly informs whether 2.9's annotations route correctly under real load.** First week of dogfood produces concrete evidence about which annotations matter.
4. **2.2 (worktree) wants real cost data first.** Per-tree budget cap design needs a baseline of single-agent runs to size the multiplier sanely. A's 7-day cost ledger is the input. **Don't ship 2.2 before A produces data.**
5. **2.3 (mutation testing) is a clean follow-up after 2.2** — the natural "Tier 1 closeout" pair is `2.2 → 2.3`, both touching verifier internals. Shipping 2.3 alone before 2.2 leaves it slightly orphaned.

## Why NOT each alternative

| Alternative ordering | Why I'd not do it |
|---|---|
| **B alone first** (worktree supervisor) | Per-tree cap design needs A's burn-in cost data; otherwise we guess at multipliers. Also blocks operator review attention for ~2 evenings on a single sprint. |
| **B + B′ together (Tier 1 closeout)** | Leaves dogfood unrun for another week+; Steward stays "theoretically safe" not "demonstrably safe". And 2.2 wants A's data first. |
| **All three (A + B + C)** | Operator attention exhausted; review pipeline competes for context. Better to land 2.9 + dogfood first, then plan 2.2 + 2.3 with real data. |
| **C alone (skip dogfood)** | Misses the chance to harden Sprint 2.4-2.8 mechanics under real overnight load. Every 2.x sprint added defenses we never observed in production conditions. |
| **A alone (skip 2.9)** | Wastes the day-time focus block while burn-in runs. Sprint 2.9 is exactly the right shape for "do something while overnight cron does its thing." |
